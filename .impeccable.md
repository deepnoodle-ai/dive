## Design Context

### Users

Dive's experimental CLI is used by engineers building, debugging, and testing
agent behavior. They work in a terminal, often inside an existing repository,
and need to understand what the agent can access, what runtime context it
received, and which actions or checks actually occurred.

### Brand Personality

Calm, precise, and transparent. The CLI should feel like a capable engineering
instrument: concise during routine work, explicit when state or authority
changes, and never dependent on model narration to explain its own mechanics.

### Aesthetic Direction

Use a quiet, terminal-native hierarchy with progressive disclosure. Keep the
normal conversation readable, surface important lifecycle events as compact
status lines, and put exact diagnostic detail behind intentional commands. Avoid
decorative noise, novelty terminology, and hidden state.

### Design Principles

1. Make boundaries visible: show the effective workspace and when it differs
   from the repository root.
2. Make runtime mechanics observable without asking the model to self-report.
3. Use exact, consistent names for reminders, authority tiers, and lifecycles.
4. Keep default output concise; expose full payloads through progressive
   disclosure.
5. Preserve automation contracts: diagnostic UI must not contaminate print or
   JSON output.
